Introduction to Carla — Topic Chat Summary

Scope of This Chat

This conversation focused specifically on:

* the role of Carla as a programmable audio/plugin host
* MIDI vs OSC control
* Carla Python integration surfaces
* .carxp project/session files
* architectural implications for declarative music systems

It did not cover the broader project philosophy around looping, performance dogma, CI/CD, or infrastructure beyond what was directly necessary to explain Carla.

⸻

What Carla Is

Carla is an open-source modular plugin host from the KX Studio ecosystem.

Important clarification:

* the correct name is Carla, not Karla

Key capabilities discussed:

* hosts VST2/VST3/AU/LV2 plugins
* routes MIDI and audio
* supports OSC remote control
* exposes modular routing graphs
* works well for programmable/live systems

Install on macOS:

brew install carla

⸻

Conceptual Model

Carla was framed as:

"a modular operating system for musical signal flow"

rather than:

"a traditional DAW"

The important distinction is that Carla foregrounds:

* nodes
* routing
* signal flow
* modular composition

rather than:

* timeline editing
* track arrangement
* linear production workflows

⸻

MIDI vs Audio

The chat distinguished clearly between:

MIDI

Symbolic performance data:

* notes
* velocity
* control changes
* timing events

MIDI is not sound.

Audio

Actual waveform/audio streams.

Typical relationship:

Keyboard → MIDI
Synth plugin → audio

Carla Plugin Control

The discussion covered:

* loading VSTs
* setting parameters
* mapping MIDI controls
* driving instruments programmatically

Illustrative Python example using:

* mido
* python-rtmidi

Example structure:

cc(CC_EXPRESSION, 100)
cc(CC_REVERB, 45)
note(60, 90, 0.5)

Key insight:

* plugin parameters are plugin-specific
* parameter IDs usually need discovery/querying
* there is no universal “expression” parameter standard

Carla Python APIs

Three relevant layers were identified.

1. Internal Python frontend/backend wrappers

Example:

source/carla_backend.py

Supports:

* add_plugin
* set_parameter_value
* query parameters
* routing operations

But treated as:

internal-ish rather than polished public SDK

2. Backend host API

Can:

* load plugins
* activate plugins
* query parameter metadata
* map MIDI CCs
* manipulate routing

3. OSC remote control

Recommended as the cleanest external integration mechanism.

⸻

OSC (Open Sound Control)

OSC was introduced as a modern network-native control protocol.

Conceptual comparison:

MIDI = older hardware-era event protocol
OSC = distributed network-native control protocol

Example OSC messages:

/reverb/mix 0.4
/strings/expression 0.8
/transport/play

Advantages discussed:

* hierarchical addresses
* human-readable semantics
* arbitrary parameter naming
* network transport
* extensibility

Important distinction:

* OSC controls systems
* OSC does not transport audio

⸻

OSC and Carla

Carla supports OSC for:

* parameter control
* transport control
* plugin activation
* automation

However:

* OSC is not considered a reliable interface for dynamically loading plugins

Recommended split:

session construction → .carxp
runtime control → OSC + MIDI

⸻

.carxp Files

The conversation converged strongly on treating .carxp as:

"deployment artefact"

rather than:

"primary authored representation"

Characteristics of .carxp

Described as:

"internal-but-readable serialised application state"

Properties:

* XML-like structured text
* human-readable
* diffable in git
* scriptable/generatable
* much cleaner than opaque document-container formats like .docx

However:

* not guaranteed stable as a public schema
* may contain plugin-specific opaque blobs
* some VST state may be serialised in proprietary forms

⸻

Recommended Architecture

Strong recommendation:

Declarative config / DSL
        ↓
Generate .carxp
        ↓
Launch Carla
        ↓
Control runtime via MIDI + OSC

Rather than:

Manually editing Carla sessions

Suggested approach:

* define a higher-level YAML-like configuration
* compile into .carxp
* keep semantic ownership outside Carla project files

Illustrative example:

plugins:
  strings:
    type: vst3
    path: /plugins/SpitfireStrings.vst3
    parameters:
      expression: 0.8
      reverb: 0.35

Compiled into:

YAML → .carxp

Carla vs Carla Rack

Two Carla modes were discussed.

Full Carla / Patchbay Mode

Supports:

* arbitrary routing graphs
* layered instruments
* multiple MIDI buses
* modular orchestration

Analogy used:

"Kubernetes for audio routing"

Recommended for this project direction.

carla-rack

Simplified linear chain mode:

plugin1 → plugin2 → plugin3

Better suited to:

* pedalboard workflows
* simple effects chains

Not ideal for complex declarative routing systems.

Overall Technical Direction

The conversation converged on an architecture where:

* Carla acts as the plugin/audio runtime
* Python acts as the declarative/control layer
* MIDI transports musical events
* OSC transports runtime control
* .carxp acts as a generated deployment/session artefact

The overall style resembles:

programmable reactive musical middleware

more than:

traditional DAW-centric music production
