# Core Artistic Dogma

## Performance should feel like music, not engineering

During performance:

- the musician interacts with pianos, keyboards, pads, pedals, expression controls, and scores;

- the audience experiences musical performance rather than software manipulation;

- the performer should not appear to be operating a DAW or editing configuration.

The engineering layer exists primarily:

- before performance;

- between performances;

- in rehearsal and experimentation;

- in infrastructure and tooling.

The live experience should remain:

- immediate;

- embodied;

- tactile;

- responsive;

- instrument-centred.


# Symbolic-First Philosophy

The project strongly prefers symbolic representations of music over rendered audio where practical.

Preferred:

- MIDI;

- notation;

- structured musical events;

- parameter automation;

- declarative configuration.

Less preferred:

- destructive audio editing;

- opaque rendered stems;

- DAW-centric workflows.

This is both:

- an artistic preference;

- an engineering preference.

Symbolic representations:

- remain editable;

- support reproducibility;

- support analysis and transformation;

- support AI tooling;

- support testing and CI.

# Practice vs Performance

The project explicitly distinguishes:

- practice workflows;

- rehearsal workflows;

- performance workflows.

## Practice mode

Practice workflows may include:

- MIDI looping;

- layered playback;

- isolated hand practice;

- slowed playback;

- generated accompaniment;

- analytical tooling.

The goal is to improve musicianship while preserving continuity of the musical soundscape.

Example:

- left hand recorded as MIDI loop;

- right hand practised live over playback.

## Performance mode

Performance mode should avoid:

- obvious engineering workflows;

- visible software manipulation;

- excessive layering complexity;

- “live coding” aesthetics unless intentionally explored later.

Performance should foreground musicianship.



# Interface Philosophy

The visible performance interface should resemble:

- an instrument rig;

- keyboards;

- pedals;

- pads;

- expression controls;

- scores or notation displays.

Not:

- mouse-driven DAW interaction;

- terminal-centric performance;

- engineering dashboards.

Engineering complexity should mostly remain hidden behind:

- presets;

- configurations;

- mappings;

- automation.

# AI Philosophy

AI is viewed as:

- an accelerator for engineering and tooling;

- a collaborator in configuration generation;

- potentially useful for musical experimentation.

But:

- musicianship remains central;

- the project is not attempting to replace musical performance with prompting;

- generated systems should remain inspectable and controllable.

The project favours:

- understandable systems;

- inspectable representations;

- compositional transparency.

# Development Philosophy

The project should be approached as:

- a serious engineering effort;

- an artistic system design problem;

- a musical instrument design exercise.

Not merely:

- a software stack;

- a DAW replacement;

- an automation experiment.

The system should ultimately disappear behind the musical experience.
